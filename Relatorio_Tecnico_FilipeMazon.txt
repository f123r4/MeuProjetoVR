LIMITAÇÃO TÉCNICA DE AMBIENTE DE DESENVOLVIMENTO

Sistema Operacional: Bazzite (baseado em Fedora Linux — distribuição imutável
orientada a jogos, derivada do rpm-ostree/SteamOS)

O projeto foi desenvolvido integralmente em Linux, sistema operacional não
suportado oficialmente pela Meta para desenvolvimento com o Meta XR SDK.
Isso gerou uma série de limitações técnicas documentadas abaixo:

1. ERRO DE INICIALIZAÇÃO DO META XR SDK (OVRProjectConfig)
   Quando abri o projeto pela primeira vez, o Unity ficava travando com um
   erro no console que dizia algo como:
     System.TypeInitializationException → ArgumentOutOfRangeException
   Demorei um bom tempo pra entender o que estava acontecendo. Pesquisando,
   descobri que no Linux a biblioteca nativa do Oculus (OVRPlugin) não
   carrega, porque não tem suporte oficial ao driver no sistema.
   Com isso, a versão do SDK ficava sendo reportada como 0.0.0, o que fazia
   um método interno (OVRProjectConfig) tentar gerar um Range com contagem
   negativa — e aí o Unity travava na inicialização.
   A solução que encontrei foi editar diretamente o arquivo do SDK dentro
   de Library/PackageCache, adicionando uma verificação antes do Range:
     .Concat(currentSdkVersion >= version2Start
         ? Enumerable.Range(version2Start, currentSdkVersion - version2Start + 1)
         : Enumerable.Empty<int>())
   O problema dessa solução é que a pasta Library não é versionada, então
   se alguém deletar ela vai precisar aplicar o patch novamente.

2. CONFLITO DO OVR HEADSET EMULATOR COM CHARACTERCONTROLLER
   Outro problema que apareceu foi o player ficando travado na porta da casa.
   O CharacterController não estava retornando isGrounded de forma consistente,
   o que causava um comportamento estranho na colisão. Aparentemente o OVR
   Headset Emulator também movia o XROrigin em paralelo com o Move() do
   CharacterController, e isso gerava conflito.
   Resolvi aumentando a abertura da porta na casa e ajustando as dimensões
   do CharacterController (Height: 1.8, Radius: 0.3, Center Y: 0.9) até
   o player conseguir passar sem travar.

3. AUSÊNCIA DO RUNTIME OCULUS/META NO LINUX
   O Linux não possui o runtime nativo do Oculus/Meta Quest (oculus-runtime),
   impossibilitando o uso do XR Device Simulator completo e a conexão USB
   com o headset Quest para testes diretos. Todo o desenvolvimento e teste
   foi realizado no Unity Editor com movimentação via teclado (WASD),
   sem emulação de controles VR.

4. SISTEMA IMUTÁVEL BAZZITE (rpm-ostree)
   O Bazzite, por ser uma distribuição imutável, não permite instalação de
   pacotes de sistema pela forma convencional (dnf install). Ferramentas de
   desenvolvimento como Android SDK, NDK e Java (necessárias para build
   Android/Meta Quest) precisam ser instaladas em camadas overlay ou via
   Flatpak/Toolbox, o que exigiu configuração adicional do ambiente.

IMPACTO NAS ENTREGAS:
   Devido às limitações acima, o projeto foi desenvolvido e testado
   exclusivamente no Unity Editor em modo PC (teclado). O build para
   Android/Meta Quest foi configurado nas Build Settings conforme os
   requisitos do projeto, mas não pôde ser testado diretamente no headset
   durante o desenvolvimento por restrições do ambiente Linux.

SEÇÃO 2 — CONCEITO DO PROJETO

2.1 Nome do Projeto: Meu Primeiro Ambiente VR

2.2 Contexto e Objetivo no Metaverso:
Criar um ambiente VR interativo de exploração onde o jogador navega por um
cenário externo com uma casa, coleta 3 objetos espalhados pelo ambiente e,
após coletar todos, acessa o interior da casa para ativar um botão interativo.
O objetivo é demonstrar os fundamentos de XR — navegação, interação por
proximidade, feedback visual via HUD e resposta a eventos do jogador —
em um projeto funcional no Unity com Meta XR SDK.

2.3 Descrição Geral do Ambiente Virtual:
Área externa gramada com um plano de chão texturizado e skybox de pôr do sol.
No centro do ambiente há uma casa construída inteiramente por script (CasaBuilder),
composta por paredes, teto, chão interno e maçaneta. Três objetos coletáveis
(Chave, Cristal e Moeda) estão posicionados ao redor da casa. O HUD flutua à
frente do jogador exibindo pontuação, contador de objetos e mensagens de evento.
Após coletar os 3 itens, a parede superior da entrada desaparece, o botão aparece
dentro da casa e o jogador pode entrar e ativá-lo — mudando sua cor de branco
para amarelo (hover) e verde (ativado).

SEÇÃO 3 — CONFIGURAÇÃO TÉCNICA DO PROJETO

3.1 Versão do Unity e Porquê:
Unity 6000.3.14f1 (Unity 6 LTS), por ser a versão estável com suporte ao
Meta XR SDK v60+ e ao Universal Render Pipeline (URP), garantindo compatibilidade
com build Android para Meta Quest.

3.2 Instalação do Meta XR SDK (Passo a Passo):
1. Abrir Window > Package Manager
2. Clicar em "+" > "Add package by name"
3. Inserir: com.meta.xr.sdk.all
4. Aguardar importação automática
O manifest.json já inclui a dependência — ao abrir o projeto ela é importada
automaticamente.

3.3 Configurações de Build para Android/Meta Quest:
- File > Build Settings > Android > Switch Platform
- Player Settings > Other Settings:
  * Minimum API Level: Android 10 (API Level 29)
  * Target API Level: Android 12 (API Level 31)
  * Texture Compression: ASTC
  * Scripting Backend: IL2CPP
  * Target Architectures: ARM64

3.4 Configuração do XR Plugin Management:
- Edit > Project Settings > XR Plugin Management
- Aba PC: marcar "Oculus" (para teste no Editor)
- Aba Android: marcar "Oculus" (para build no Quest)

3.5 Movimentação no PC (Editor):
Movimentação via teclado (WASD / setas) usando CharacterController com gravidade,
implementado no PlayerController.cs. O CharacterController é configurado com
Height: 1.8, Radius: 0.3 e Center Y: 0.9, garantindo ancoragem correta ao chão.
Controles: W/S = frente/trás | A/D = esquerda/direita | E ou clique = interagir.

SEÇÃO 7 — COMO O PROJETO FOI DESENVOLVIDO

Primeiro instalei o Unity 6000.3.14f1 pelo Hub e criei o projeto 3D com URP.
Depois tentei instalar o Meta XR SDK, que é quando apareceu o primeiro erro
(descrito na Seção 1). Levou um tempo até conseguir abrir o projeto sem crashes.

Com o projeto abrindo, configurei o XR Plugin Management (Oculus no PC e Android)
e ajustei as Build Settings pro Android com API 29, ASTC, IL2CPP e ARM64.

Comecei a cena com um Plane pra servir de chão e fui construindo a casa. A ideia
de fazer um script pra gerar as paredes (CasaBuilder.cs) veio porque ficava muito
chato ajustar posição e escala de cubo por cubo no editor. Com o script ficou
muito mais fácil de refazer quando precisava mudar alguma coisa.

Com a cena base pronta, fiz o PlayerController com movimentação WASD e gravidade
via CharacterController. Aí apareceu o segundo problema — o player ficava travando
na porta da casa, que resolvi ajustando as dimensões do CharacterController e
aumentando a abertura (detalhado na Seção 1).

Os objetos coletáveis e o sistema de pontuação foram feitos em seguida. Separei
em Controller e View pra cada objeto, seguindo o padrão MVC que o projeto de
referência usava. O GameManager ficou responsável por controlar o estado geral —
pontuação, quando o botão aparece, e quando a parede da entrada some.

O HUD foi uma parte que tive que ajustar bastante. No começo o canvas ficava
parado na cena e não seguia o player. Resolvi usando LateUpdate pra posicionar
o canvas sempre na frente da câmera.

Por fim testei o fluxo completo: aproximar dos objetos pra coletar, ver o botão
aparecer, entrar na casa e pressionar E pra ativar. Ajustei as posições dos
objetos e do botão pra ficarem em lugares que fizessem sentido no cenário.

SEÇÃO 8 — REFLEXÃO FINAL

8.1 Aprendizado:
O que mais aprendi foi a separar o código em partes com responsabilidade diferente.
No começo eu colocava tudo no GameManager e ficava um arquivo gigante difícil de
mexer. Quando separei em Controller e View pra cada objeto, ficou muito mais fácil
de entender o que cada parte fazia. Também aprendi que o Meta XR SDK tem bastante
coisa não documentada — o erro no Linux levou um bom tempo pra resolver porque não
tinha ninguém com o mesmo problema em forum. Outra coisa foi os scripts de Editor
(CasaBuilder etc.), que não conhecia e me pouparam muito trabalho de ajuste manual.

8.2 Dificuldades Encontradas:
A maior dificuldade foi o Linux. Não sabia que o SDK não tinha suporte oficial e
fui descobrindo isso conforme os erros apareciam. O patch no PackageCache funciona
mas é chato porque some quando a pasta Library é deletada — não é uma solução ideal.
Testar sem headset também foi limitado, não sei se a navegação VR real vai funcionar
bem até conseguir testar num Quest de verdade. Tem partes do SDK que ainda não
entendo direito, especialmente a parte de XR Interaction Toolkit.

8.3 Possíveis melhorias:
- Colocar sons quando o player coleta um objeto ou aperta o botão. Tentei um
  AudioSource mas não deu tempo de ajustar direito.
- Os objetos coletáveis são cubos e esferas primitivas. Com mais tempo colocaria
  modelos 3D mais interessantes no lugar.
- Testar no headset Quest quando tiver acesso a um ambiente Windows ou Mac
  pra não ter problema com o SDK.
- Talvez expandir pra ter mais de uma sala ou mais objetos pra coletar.
